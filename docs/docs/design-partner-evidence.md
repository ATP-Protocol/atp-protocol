# Design Partner Evidence

A design partner note should prove one governed agent action, not describe a broad aspiration.

## Case note template

```markdown
# ATP Design Partner Case Note

## Context

- Organization/team:
- Environment:
- Agent/client:
- Tool/action:

## Risk

- What could go wrong:
- Required policy:
- Required approval:
- Credential provider and scope:

## ATP Contract

- Contract ID:
- Authority:
- Action:
- Attestation:
- Approval rule:

## Proof Run

- Date:
- ATP version:
- Gateway/version:
- Conformance level:
- Evidence IDs:
- Outcome summary:

## Result

- Allowed scenario:
- Denied scenario:
- Pending/approval scenario:
- Operator or auditor feedback:

## Follow-up

- Production blockers:
- Required integrations:
- Security review notes:
```

## Minimum bar

A note is strong enough to count when it includes:

- one specific governed action;
- one allowed execution;
- one denied execution;
- one approval or pending-review path for a higher-risk action;
- at least one evidence ID;
- the policy reason for the denial;
- the role or identity that has approval authority.

## Public format

If a partner cannot be named, publish an anonymized report with the organization category, risk class, tool type, and proof results. Do not publish secrets, private endpoints, customer data, or raw payloads.
